# บทที่ 4.1 — หลักการความปลอดภัยข้อมูล
**(2 ชั่วโมง)**

---

## 🔐 ทำไมความปลอดภัยข้อมูลถึงสำคัญ?

ลองนึกถึงสถานการณ์เหล่านี้:

> 🏦 **เหตุการณ์ที่ 1 — ธนาคาร**
> แฮกเกอร์เจาะระบบธนาคารออนไลน์ได้สำเร็จ
> ข้อมูลบัตรเครดิตลูกค้า 2 ล้านรายถูกขโมย
> ธนาคารต้องจ่ายค่าชดเชยและค่าปรับรวม **1,200 ล้านบาท**
> ชื่อเสียงเสียหาย ลูกค้าย้ายธนาคารกว่า 30%

> 🏥 **เหตุการณ์ที่ 2 — โรงพยาบาล**
> ระบบ IT ถูก Ransomware (แรนซัมแวร์) โจมตี
> ข้อมูลผู้ป่วยทั้งหมดถูกเข้ารหัสจนอ่านไม่ได้
> แพทย์ไม่สามารถดูประวัติการรักษาได้
> ต้องเลื่อนการผ่าตัดฉุกเฉิน → ผู้ป่วยเสี่ยงอันตราย

> 🏫 **เหตุการณ์ที่ 3 — โรงเรียน**
> นักเรียนคนหนึ่งล็อกอินเข้าระบบด้วยบัญชีครู
> แก้ไขผลการเรียนของตัวเองและเพื่อน
> ข้อมูลเกรดทั้งปีเสียหาย สอบสวนได้ยากมากเพราะไม่มี Log

**บทเรียน:** ข้อมูลที่ไม่มีการรักษาความปลอดภัย = ทรัพย์สินที่ไม่มีกุญแจล็อก

---

## 🛡️ CIA Triad — สามเหลี่ยมความปลอดภัยข้อมูล

**CIA Triad** (ซีไอเอ ไทรแอด) คือหลักการพื้นฐาน 3 ด้านของการรักษาความปลอดภัยข้อมูล ที่ทุกระบบต้องคำนึงถึง

```
              🔒 Confidentiality
             (คอนฟิเดนเชียลลิตี้)
                  "ความลับ"
                    /\
                   /  \
                  /    \
                 /      \
                /________\
🔁 Integrity              ✅ Availability
(อินทิกริตี้)           (อะเวลละบิลลิตี้)
"ความถูกต้อง"           "ความพร้อมใช้"
```

---

### 🔒 C — Confidentiality (คอนฟิเดนเชียลลิตี้) — "ความลับ"

**ความหมาย:** ข้อมูลต้องถูกเข้าถึงได้เฉพาะผู้ที่มีสิทธิ์เท่านั้น ป้องกันการเข้าถึงโดยไม่ได้รับอนุญาต

> **เปรียบเทียบ:** เหมือนห้องนิรภัยธนาคาร — เข้าได้เฉพาะพนักงานที่มีกุญแจและรหัสเท่านั้น

**ตัวอย่างในชีวิตจริง:**

| สถานการณ์ | ปัญหา Confidentiality | วิธีป้องกัน |
|-----------|----------------------|------------|
| พนักงานทั่วไปเห็นเงินเดือนผู้บริหาร | เปิดเผยข้อมูลลับ | กำหนด Role แยกกัน |
| นักเรียนเห็นคำตอบข้อสอบก่อนสอบ | ข้อมูลรั่วไหล | เข้ารหัสไฟล์ข้อสอบ |
| ข้อมูลอุบัติเหตุมีชื่อผู้ประสบเหตุ เปิดให้สาธารณะดูได้ทั้งหมด | ละเมิดความเป็นส่วนตัว | ซ่อน/ลบข้อมูลส่วนบุคคล |

**เทคนิคป้องกัน:**
- **การเข้ารหัส (Encryption)** — ทำให้ข้อมูลอ่านไม่ออกหากไม่มีกุญแจ
- **การตั้งรหัสผ่าน** — ยืนยันตัวตนก่อนเข้าถึง
- **การกำหนดสิทธิ์ (Access Control)** — แต่ละคนเห็นได้แค่ส่วนที่เกี่ยวข้องกับตน

---

### 🔁 I — Integrity (อินทิกริตี้) — "ความถูกต้องสมบูรณ์"

**ความหมาย:** ข้อมูลต้องถูกต้อง ไม่ถูกแก้ไขหรือทำลายโดยไม่ได้รับอนุญาต

> **เปรียบเทียบ:** เหมือนเอกสารราชการที่มีตราประทับ — ถ้ามีการแก้ไขจะรู้ทันที

**ตัวอย่างในชีวิตจริง:**

| สถานการณ์ | ปัญหา Integrity | ผลกระทบ |
|-----------|----------------|---------|
| นักเรียนแก้เกรดตัวเองในระบบ | ข้อมูลถูกดัดแปลง | เกรดไม่สะท้อนความเป็นจริง |
| ไวรัสเปลี่ยนตัวเลขในไฟล์บัญชี | ข้อมูลการเงินผิดพลาด | รายงานงบการเงินผิด |
| อุบัติเหตุถูกบันทึกผิดประเภทโดยเจตนา | บิดเบือนสถิติ | นโยบายความปลอดภัยถนนผิดพลาด |

**เทคนิคป้องกัน:**
- **Audit Log** — บันทึกว่าใครแก้ไขอะไร เมื่อไหร่
- **Checksum / Hash** — ตรวจสอบว่าไฟล์ถูกดัดแปลงหรือไม่
- **Database Constraint** — ป้องกันข้อมูลผิดรูปแบบเข้าสู่ระบบ

---

### ✅ A — Availability (อะเวลละบิลลิตี้) — "ความพร้อมใช้งาน"

**ความหมาย:** ระบบและข้อมูลต้องพร้อมใช้งานได้เมื่อต้องการ ไม่หยุดชะงัก

> **เปรียบเทียบ:** เหมือนไฟฟ้าในบ้าน — ต้องใช้ได้ตลอด 24 ชั่วโมง ถ้าดับก็คือปัญหา

**ตัวอย่างในชีวิตจริง:**

| สถานการณ์ | ปัญหา Availability | ผลกระทบ |
|-----------|--------------------|---------|
| ระบบโรงพยาบาลล่มระหว่างฉุกเฉิน | เข้าถึงข้อมูลผู้ป่วยไม่ได้ | เสี่ยงชีวิตผู้ป่วย |
| Server ของโรงเรียนพัง วันส่งผลเรียน | ข้อมูลเข้าถึงไม่ได้ | งานหยุดชะงัก |
| ไฟล์สถิติอุบัติเหตุถูกลบโดยไม่ตั้งใจ | ข้อมูลหายถาวร | วิเคราะห์นโยบายไม่ได้ |

**เทคนิคป้องกัน:**
- **Backup (สำรองข้อมูล)** — มีข้อมูลสำรองพร้อมกู้คืน
- **Redundancy** — มีระบบสำรอง (Backup Server)
- **UPS (Uninterruptible Power Supply)** — ป้องกันไฟดับกะทันหัน

---

## ⚖️ กฎหมาย PDPA — พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562

**PDPA** (พีดีพีเอ) ย่อมาจาก **Personal Data Protection Act** (เพอร์ซันนอล เดต้า โพรเทคชัน แอคท์) = พระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล

> **ใครต้องปฏิบัติตาม?** ทุกองค์กร ทุกธุรกิจ ทุกโรงเรียน ที่เก็บข้อมูลส่วนบุคคลของคนไทย

---

### ข้อมูลส่วนบุคคลคืออะไร?

**ข้อมูลส่วนบุคคล (Personal Data / เพอร์ซันนอล เดต้า)** คือข้อมูลที่สามารถระบุตัวตนของบุคคลได้ ไม่ว่าทางตรงหรือทางอ้อม

```
ตัวอย่างข้อมูลส่วนบุคคล:

📌 ระบุตัวตนโดยตรง:        📌 ระบุตัวตนโดยอ้อม:
   - ชื่อ-นามสกุล              - อายุ + จังหวัด + อาชีพ
   - เลขบัตรประชาชน            - ละติจูด + ลองจิจูด (พิกัด GPS)
   - หมายเลขโทรศัพท์           - IP Address
   - อีเมล                    - ประวัติการซื้อ
   - รูปถ่าย                  - พฤติกรรมการใช้แอป
```

**ในข้อมูลอุบัติเหตุของเรา:**

| คอลัมน์ | ประเภท | ต้องระวัง? |
|---------|--------|-----------|
| `อายุ_ปี`, `เพศ`, `อาชีพ` | ข้อมูลส่วนบุคคล (ทางอ้อม) | ⚠️ ใช่ |
| `ละติจูด`, `ลองจิจูด` | ระบุพิกัดสถานที่ | ⚠️ ใช่ |
| `จังหวัด`, `ภูมิภาค` | ข้อมูลทั่วไป | ✅ ไม่เป็นไร |
| `จำนวนผู้เสียชีวิต` | สถิติรวม | ✅ ไม่เป็นไร |

---

### สิทธิ์ของเจ้าของข้อมูลตาม PDPA (5 สิทธิ์หลัก)

| สิทธิ์ | ความหมาย | ตัวอย่าง |
|--------|----------|---------|
| **Right to Access** | ขอดูข้อมูลของตัวเองได้ | ขอดูว่าโรงเรียนเก็บข้อมูลอะไรเราบ้าง |
| **Right to Rectification** | ขอแก้ไขข้อมูลที่ผิดได้ | แจ้งให้แก้ชื่อที่สะกดผิด |
| **Right to Erasure** | ขอลบข้อมูลได้ ("สิทธิ์ถูกลืม") | ขอให้ลบประวัติออกจากระบบ |
| **Right to Object** | คัดค้านการนำข้อมูลไปใช้ได้ | ปฏิเสธไม่ให้ใช้ข้อมูลส่งโฆษณา |
| **Right to Portability** | ขอรับข้อมูลของตัวเองได้ | ขอ Export ข้อมูลตัวเองเป็นไฟล์ |

---

### โทษหากฝ่าฝืน PDPA

| ประเภทโทษ | ระดับ | ตัวอย่าง |
|-----------|-------|---------|
| **โทษทางแพ่ง** | ชดใช้ค่าเสียหาย | จ่ายค่าชดเชยแก่เจ้าของข้อมูล |
| **โทษทางปกครอง** | ปรับสูงสุด 5 ล้านบาท | รั่วไหลข้อมูลโดยไม่แจ้ง |
| **โทษทางอาญา** | จำคุกสูงสุด 1 ปี + ปรับ 1 ล้านบาท | นำข้อมูลไปแสวงหาประโยชน์โดยไม่ได้รับอนุญาต |

> **💡 ในฐานะนักเรียน:** แม้ไม่ได้เป็นองค์กร แต่ถ้าเราพัฒนาระบบหรือทำ Project ที่เก็บข้อมูลส่วนบุคคลของผู้อื่น ก็ต้องปฏิบัติตาม PDPA ด้วย!

# บทที่ 4.2 — การควบคุมการเข้าถึงข้อมูล
**(3 ชั่วโมง)**

---

## 👤 แนวคิด User, Role, Permission

ระบบฐานข้อมูลที่ดีต้องควบคุมได้ว่า "ใครเข้าถึงอะไรได้บ้าง"

```
ระบบควบคุมการเข้าถึง (Access Control System):

👤 USER            📋 ROLE              🔑 PERMISSION
(ผู้ใช้งาน)  →   (บทบาท/กลุ่ม)  →   (สิทธิ์ที่ได้รับ)

ตัวอย่างโรงเรียน:
สมชาย     →   ครูวิทย์      →   SELECT, INSERT ตารางเกรด
สมหญิง    →   นักเรียน      →   SELECT เฉพาะเกรดตัวเอง
admin     →   ผู้ดูแลระบบ   →   ทุกสิทธิ์ทุกตาราง
```

**ทำไมต้องแยก Role?**

> **เปรียบเทียบ:** โรงพยาบาลมีบัตรเข้าประตูแต่ละชั้น
> - พยาบาล → เข้าได้แค่ชั้นผู้ป่วยที่ตนดูแล
> - แพทย์ → เข้าได้ทุกชั้น
> - แม่บ้าน → เข้าได้แค่ห้องที่ต้องทำความสะอาด
> - ผู้เยี่ยม → เข้าได้แค่ห้องโถง
> ถ้าทุกคนมีบัตรเหมือนกัน = ระบบความปลอดภัยพัง!

---

## 🔑 Permission ประเภทต่าง ๆ ใน Database

| Permission | คำอ่าน | ความหมาย | ตัวอย่างการใช้ |
|-----------|--------|----------|--------------|
| `SELECT` | ซีเล็กต์ | อ่านข้อมูลได้ | นักเรียนดูเกรดตัวเอง |
| `INSERT` | อินเสิร์ท | เพิ่มข้อมูลได้ | ครูบันทึกคะแนน |
| `UPDATE` | อัปเดท | แก้ไขข้อมูลได้ | ครูแก้คะแนนที่บันทึกผิด |
| `DELETE` | ดีลีท | ลบข้อมูลได้ | ผู้ดูแลลบข้อมูลเก่า |
| `CREATE` | ครีเอท | สร้างตาราง/ฐานข้อมูลได้ | DBA เท่านั้น |
| `DROP` | ดรอป | ลบตาราง/ฐานข้อมูลได้ | DBA เท่านั้น |
| `ALL PRIVILEGES` | ออล พริวิเลจ | ทุกสิทธิ์ | Admin เท่านั้น |

---

## 🛠️ คำสั่ง GRANT และ REVOKE ใน MySQL

### GRANT — มอบสิทธิ์

**รูปแบบ:**
```sql
GRANT permission ON database.table TO 'username'@'host';
```

**ตัวอย่างระบบโรงเรียน:**

```sql
-- สร้าง User ครูก่อน
CREATE USER 'teacher_somsak'@'localhost' IDENTIFIED BY 'T3ach3r#2026';

-- ให้ครูอ่านและบันทึกคะแนนได้
GRANT SELECT, INSERT, UPDATE ON school_db.grades TO 'teacher_somsak'@'localhost';

-- ให้ครูดูรายชื่อนักเรียนได้ (แต่แก้ไขไม่ได้)
GRANT SELECT ON school_db.students TO 'teacher_somsak'@'localhost';

-- ใช้งานสิทธิ์ทันที
FLUSH PRIVILEGES;
```

```sql
-- สร้าง User นักเรียน
CREATE USER 'student_malee'@'localhost' IDENTIFIED BY 'St4d3nt#2026';

-- ให้นักเรียนอ่านได้เฉพาะตารางที่ไม่ลับ
GRANT SELECT ON school_db.subjects TO 'student_malee'@'localhost';
GRANT SELECT ON school_db.schedule TO 'student_malee'@'localhost';

-- ห้ามเข้าตารางเกรดของคนอื่น (ไม่ GRANT ตาราง grades)
FLUSH PRIVILEGES;
```

```sql
-- ผู้ดูแลระบบมีทุกสิทธิ์
CREATE USER 'admin_it'@'localhost' IDENTIFIED BY 'Adm1n#Sup3r!';
GRANT ALL PRIVILEGES ON school_db.* TO 'admin_it'@'localhost';
FLUSH PRIVILEGES;
```

### REVOKE — ถอนสิทธิ์

**รูปแบบ:**
```sql
REVOKE permission ON database.table FROM 'username'@'host';
```

**ตัวอย่าง:**

```sql
-- ถอนสิทธิ์ลบข้อมูลจากครู (ครูไม่ควรลบเกรดได้)
REVOKE DELETE ON school_db.grades FROM 'teacher_somsak'@'localhost';

-- ถอนทุกสิทธิ์จาก User ที่ลาออกแล้ว
REVOKE ALL PRIVILEGES ON school_db.* FROM 'teacher_somsak'@'localhost';

-- ลบ User ออกจากระบบ
DROP USER 'teacher_somsak'@'localhost';

FLUSH PRIVILEGES;
```

## 🔒 Row-Level Security — ควบคุมระดับแถว

บางครั้งไม่ใช่แค่ "เข้าตารางได้ไหม" แต่ต้องควบคุมว่า "เห็นได้แค่แถวไหน"

**ตัวอย่าง:** นักเรียนเข้าตาราง `grades` ได้ แต่เห็นเฉพาะเกรดตัวเองเท่านั้น

```sql
-- สร้าง VIEW สำหรับนักเรียนแต่ละคน
-- (VIEW คือ "มุมมอง" ที่กรองข้อมูลแล้ว)

CREATE VIEW my_grades AS
SELECT g.subject_id, s.subject_name, g.score, g.grade
FROM grades g
JOIN subjects s ON g.subject_id = s.subject_id
WHERE g.student_id = CURRENT_USER();  -- เห็นเฉพาะของตัวเอง

-- ให้นักเรียนเข้าถึง VIEW ได้แทนตารางโดยตรง
GRANT SELECT ON school_db.my_grades TO 'student_malee'@'localhost';
-- ห้ามเข้าตาราง grades โดยตรง (ไม่ GRANT)
```

**ผลลัพธ์:**
- `student_malee` เห็นได้เฉพาะเกรดของตัวเอง
- ถึงจะพยายาม `SELECT * FROM grades` ก็จะได้ Error: Permission denied

## ⚠️ ความเสี่ยงของข้อมูลคืออะไร?

**ความเสี่ยง (Risk / ริสก์)** = โอกาสที่เหตุการณ์ที่ไม่ดีจะเกิดขึ้น × ผลกระทบที่ตามมา

> **เปรียบเทียบ:** เหมือนการขับรถ
> - ขับรถบนทางด่วนตอนฝนตกหนัก → โอกาสลื่นสูง + ผลกระทบรุนแรง = **ความเสี่ยงสูงมาก**
> - ขับรถในซอยบ้านตอนฟ้าแจ่ม → โอกาสเกิดเหตุต่ำ + ผลกระทบน้อย = **ความเสี่ยงต่ำ**

---

## 📂 ประเภทของความเสี่ยงข้อมูล

### ประเภทที่ 1 — Data Risk (เดต้า ริสก์) — "ความเสี่ยงจากตัวข้อมูลเอง"

ความเสี่ยงที่เกิดจากสภาพของข้อมูลหรือวิธีจัดเก็บ

| ความเสี่ยง | ตัวอย่าง | ผลกระทบ |
|-----------|---------|---------|
| **ข้อมูลสูญหาย** | ไฟล์ถูกลบโดยไม่ตั้งใจ / Hard Disk พัง | วิเคราะห์ไม่ได้ ต้องเก็บใหม่ |
| **ข้อมูลรั่วไหล** | แฮกเกอร์ขโมยไฟล์ CSV ที่มีข้อมูลผู้ป่วย | ละเมิด PDPA โทษหนัก |
| **ข้อมูลเสียหาย** | ไฟล์ corrupt ระหว่าง Transfer | ข้อมูลอ่านไม่ออก |
| **ข้อมูลล้าสมัย** | ใช้แผนที่ถนนปี 2010 วิเคราะห์อุบัติเหตุ 2568 | บทสรุปผิดพลาด |

### ประเภทที่ 2 — Usage Risk (ยูสเซจ ริสก์) — "ความเสี่ยงจากการใช้ข้อมูล"

ความเสี่ยงที่เกิดจากการนำข้อมูลไปใช้ผิดวัตถุประสงค์

| ความเสี่ยง | ตัวอย่าง | ผลกระทบ |
|-----------|---------|---------|
| **ใช้ข้อมูลผิดวัตถุประสงค์** | นำข้อมูลอุบัติเหตุไปขายบริษัทประกัน | ผิด PDPA, จริยธรรม |
| **วิเคราะห์ผิดพลาด** | ข้อมูลมี Bias แต่ไม่รู้ → สรุปผิด | นโยบายผิดทิศ |
| **เข้าถึงโดยไม่มีสิทธิ์** | นักเรียนเห็นเกรดเพื่อน | ละเมิดความเป็นส่วนตัว |
| **SQL Injection** | แฮกเกอร์ใส่คำสั่งพิเศษผ่านช่อง Input | ขโมยหรือลบข้อมูลทั้งหมด |

---

## 📊 Risk Matrix — ตารางประเมินความเสี่ยง

**Risk Matrix** (ริสก์ เมทริกซ์) คือเครื่องมือประเมินความเสี่ยงโดยพิจารณา 2 มิติ:
- **ความน่าจะเป็น (Probability / โพรแบบิลลิตี้)** — โอกาสที่จะเกิดขึ้น
- **ผลกระทบ (Impact / อิมแพกท์)** — ความรุนแรงถ้าเกิดขึ้นจริง

```
                    ผลกระทบ (Impact)
                ต่ำ      กลาง     สูง     วิกฤต
              ───────────────────────────────────
        สูง  │  🟡      🟠       🔴      🔴
ความ    กลาง │  🟢      🟡       🟠      🔴
น่าจะ   ต่ำ  │  🟢      🟢       🟡      🟠
เป็น    ต่ำมาก│  🟢      🟢       🟢      🟡
              ───────────────────────────────────

🟢 ต่ำ (Low)      = ยอมรับได้ / เฝ้าระวัง
🟡 ปานกลาง (Med) = ต้องมีแผนจัดการ
🟠 สูง (High)     = ต้องดำเนินการเร่งด่วน
🔴 วิกฤต (Critical)= หยุดระบบแล้วแก้ทันที
```

**สูตร:** `ระดับความเสี่ยง = ความน่าจะเป็น × ผลกระทบ`

---

## 📋 ตัวอย่างการประเมินความเสี่ยง — ระบบข้อมูลโรงเรียน

| ความเสี่ยง | ความน่าจะเป็น (1-5) | ผลกระทบ (1-5) | คะแนนรวม | ระดับ | แนวทางจัดการ |
|-----------|---------------------|--------------|---------|-------|------------|
| ไฟล์ฐานข้อมูลถูกลบโดยไม่ตั้งใจ | 3 | 5 | **15** | 🔴 วิกฤต | Backup ทุกวัน, ยืนยัน 2 ขั้นตอนก่อนลบ |
| รหัสผ่าน Admin อ่อนแอเกินไป | 4 | 5 | **20** | 🔴 วิกฤต | บังคับใช้รหัสผ่านซับซ้อน, เปลี่ยนทุก 90 วัน |
| นักเรียนแอบเห็นเกรดเพื่อน | 3 | 3 | **9** | 🟠 สูง | Row-Level Security, Audit Log |
| ไฟฟ้าดับทำให้ข้อมูลหาย | 2 | 4 | **8** | 🟠 สูง | UPS + Auto-save ทุก 5 นาที |
| ครูลืม Logout จาก PC สาธารณะ | 4 | 2 | **8** | 🟠 สูง | Auto-logout หลัง 10 นาที |
| มี SQL Injection ผ่านเว็บ | 2 | 5 | **10** | 🔴 วิกฤต | Parameterized Query ทุก Query |
| ไฟล์ CSV ส่งทางอีเมลไม่เข้ารหัส | 3 | 3 | **9** | 🟠 สูง | เข้ารหัสไฟล์ก่อนส่ง |
| Hard Disk เสียหาย | 1 | 5 | **5** | 🟡 ปานกลาง | Backup ไว้ที่อื่น (Cloud / USB) |

```
ขั้น 1: ระบุทรัพย์สินข้อมูล (What do we protect?)

        ↓
```

## 🔐 การเข้ารหัส (Encryption / เอ็นคริปชัน)

**Encryption** (เอ็นคริปชัน) คือกระบวนการแปลงข้อมูลให้อ่านไม่ออก เว้นแต่ผู้ที่มี "กุญแจ" ถอดรหัส

```
ข้อความปกติ:  "รหัสผ่าน: 1234"
        ↓  เข้ารหัส (Encrypt)
ข้อมูลรหัส:  "a3f8b2c9e7d1..."
        ↓  ถอดรหัส (Decrypt) ด้วย Key
ข้อความปกติ:  "รหัสผ่าน: 1234"
```

---

### Hash Function — ฟังก์ชันทางเดียว

**Hash** (แฮช) คือการเข้ารหัสแบบ "ทางเดียว" (One-way) — แปลงข้อมูลได้ แต่ **ย้อนกลับไม่ได้!**

> **เปรียบเทียบ:** เหมือนการบดเนื้อ → บดแล้วได้เนื้อบด แต่เอาเนื้อบดกลับมาเป็นเนื้อชิ้นเดิมไม่ได้
> ใช้สำหรับ Password เพราะระบบไม่จำเป็นต้อง "รู้" รหัสผ่านจริง แค่เปรียบเทียบ Hash ก็พอ

**กระบวนการเก็บ Password ที่ถูกต้อง:**

```
ผู้ใช้ตั้ง Password "MyPass123"
    ↓  Hash
เก็บในฐานข้อมูล: "ef92b778ba50a3b..."  ← ไม่ใช่ Password จริง!

ผู้ใช้ Login พิมพ์ "MyPass123"
    ↓  Hash อีกครั้ง
ได้: "ef92b778ba50a3b..."
    ↓  เปรียบเทียบกับที่เก็บไว้
ตรงกัน → Login สำเร็จ! ✅
```

---

### 🐍 Hash Password ด้วย Python `hashlib`

```python
import hashlib
import os

# ============================
# วิธีที่ 1: SHA-256 พื้นฐาน
# ============================

password = "MyPassword123"

# สร้าง Hash
hash_object = hashlib.sha256(password.encode('utf-8'))
hashed_password = hash_object.hexdigest()

print(f"Password เดิม: {password}")
print(f"SHA-256 Hash:  {hashed_password}")
# Output: ef92b778bafe772978827e105d3afe369da3a6dc...

# ทดสอบว่า Hash เหมือนกันไหม
test_password = "MyPassword123"
test_hash = hashlib.sha256(test_password.encode('utf-8')).hexdigest()

if test_hash == hashed_password:
    print("✅ รหัสผ่านถูกต้อง!")
else:
    print("❌ รหัสผ่านไม่ถูกต้อง!")
```

```python
# ============================
# วิธีที่ 2: เพิ่ม Salt (ซอลท์) — ป้องกันการโจมตีด้วย Rainbow Table
# ============================

# Salt คือข้อมูลสุ่มที่ใส่เพิ่มเข้าไปก่อน Hash
# เพื่อให้ Hash ของรหัสเดียวกันไม่เหมือนกัน

def hash_password_with_salt(password: str) -> tuple:
    """สร้าง Hash พร้อม Salt"""
    # สร้าง Salt แบบสุ่ม 16 bytes
    salt = os.urandom(16)

    # รวม salt + password แล้ว Hash
    salted = salt + password.encode('utf-8')
    hashed = hashlib.sha256(salted).hexdigest()

    # คืนค่า salt (hex) และ hash สำหรับเก็บในฐานข้อมูล
    return salt.hex(), hashed

def verify_password(password: str, salt_hex: str, stored_hash: str) -> bool:
    """ตรวจสอบว่ารหัสผ่านถูกต้องไหม"""
    salt = bytes.fromhex(salt_hex)
    salted = salt + password.encode('utf-8')
    test_hash = hashlib.sha256(salted).hexdigest()
    return test_hash == stored_hash

# ===== ทดสอบ =====
password = "SomchaiPass2026"

# ตอนสมัครสมาชิก
salt, hashed = hash_password_with_salt(password)
print(f"เก็บในฐานข้อมูล:")
print(f"  Salt:  {salt}")
print(f"  Hash:  {hashed}")

# ตอน Login — ถูกต้อง
result = verify_password("SomchaiPass2026", salt, hashed)
print(f"\nLogin 'SomchaiPass2026': {'✅ สำเร็จ' if result else '❌ ล้มเหลว'}")

# ตอน Login — ผิด
result2 = verify_password("WrongPassword", salt, hashed)
print(f"Login 'WrongPassword':   {'✅ สำเร็จ' if result2 else '❌ ล้มเหลว'}")
```

**ผลลัพธ์:**
```
เก็บในฐานข้อมูล:
  Salt:  3a7f8b2c...
  Hash:  d9e4a1b2...

Login 'SomchaiPass2026': ✅ สำเร็จ
Login 'WrongPassword':   ❌ ล้มเหลว
```

> **⚠️ ข้อควรรู้:** ในระบบจริงควรใช้ `bcrypt` หรือ `argon2` แทน SHA-256 เพราะซับซ้อนและปลอดภัยกว่า แต่สำหรับการเรียนรู้ SHA-256 เพียงพอ

## 💾 กลยุทธ์ 3-2-1 Backup

**กลยุทธ์ 3-2-1** คือมาตรฐานสากลของการสำรองข้อมูลที่ดีที่สุด:

```
3 — เก็บข้อมูล 3 ชุด
    (ต้นฉบับ 1 + สำรอง 2)

2 — เก็บไว้บน 2 ประเภทสื่อต่างกัน
    (เช่น Hard Disk + USB Flash Drive)

1 — เก็บไว้ 1 ชุดนอกสถานที่
    (เช่น Cloud หรือบ้านอีกหลัง)
```

**ตัวอย่างระบบโรงเรียน:**

```
📁 ฐานข้อมูลหลัก (school_data.db)
   ├── 💻 บน PC ห้อง Server โรงเรียน       ← ชุดที่ 1 (ต้นฉบับ)
   ├── 💽 External Hard Disk ในห้อง IT      ← ชุดที่ 2 (ต่างประเภทสื่อ)
   └── ☁️ Google Drive / OneDrive           ← ชุดที่ 3 (นอกสถานที่)

กำหนดการ Backup:
   - ทุกวัน: Backup ไปยัง External Hard Disk (Incremental)
   - ทุกสัปดาห์: Backup เต็ม (Full Backup) ไป Cloud
   - ทุกเดือน: ตรวจสอบว่ากู้คืนได้จริง (Test Restore)
```

**ทำ Backup อัตโนมัติด้วย Python:**

```python
import shutil
import os
from datetime import datetime

def backup_database(source_db: str, backup_folder: str):
    """
    สำรองข้อมูลฐานข้อมูล SQLite
    source_db: ไฟล์ฐานข้อมูลต้นฉบับ
    backup_folder: โฟลเดอร์เก็บ Backup
    """

    # สร้างโฟลเดอร์ถ้ายังไม่มี
    os.makedirs(backup_folder, exist_ok=True)

    # ชื่อไฟล์ Backup รวมวันที่-เวลา
    timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
    backup_filename = f"school_data_backup_{timestamp}.db"
    backup_path = os.path.join(backup_folder, backup_filename)

    # คัดลอกไฟล์
    shutil.copy2(source_db, backup_path)

    # ตรวจสอบขนาดไฟล์
    size_mb = os.path.getsize(backup_path) / (1024 * 1024)

    print(f"✅ Backup สำเร็จ!")
    print(f"   ไฟล์: {backup_filename}")
    print(f"   ขนาด: {size_mb:.2f} MB")
    print(f"   เวลา: {datetime.now().strftime('%d/%m/%Y %H:%M:%S')}")

    return backup_path

# ===== ใช้งาน =====
backup_database(
    source_db='school_data.db',
    backup_folder='D:/backups/school/'
)
```

## 📝 Audit Log — บันทึกการเข้าถึงข้อมูล

**Audit Log** (ออดิท ล็อก) คือบันทึกประวัติการกระทำในระบบ ว่าใครทำอะไร เมื่อไหร่ และผลลัพธ์คืออะไร

> **เปรียบเทียบ:** เหมือนกล้องวงจรปิดในธนาคาร ถ้ามีเหตุการณ์ผิดปกติสามารถย้อนดูได้

**ข้อมูลที่ควรบันทึกใน Audit Log:**

```sql
-- สร้างตาราง Audit Log
CREATE TABLE audit_log (
    log_id       INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp    DATETIME DEFAULT CURRENT_TIMESTAMP,  -- เวลาที่เกิดเหตุ
    username     TEXT NOT NULL,                        -- ใคร
    action       TEXT NOT NULL,                        -- ทำอะไร (SELECT/INSERT/UPDATE/DELETE)
    table_name   TEXT,                                 -- ตารางไหน
    record_id    TEXT,                                 -- record อะไร
    old_value    TEXT,                                 -- ค่าเดิม
    new_value    TEXT,                                 -- ค่าใหม่
    ip_address   TEXT,                                 -- มาจาก IP อะไร
    status       TEXT DEFAULT 'SUCCESS'                -- สำเร็จหรือไม่
);
```

**ตัวอย่าง Audit Log ที่ดี:**

| timestamp | username | action | table_name | record_id | old_value | new_value | status |
|-----------|----------|--------|-----------|-----------|-----------|-----------|--------|
| 2568-11-01 08:30:15 | teacher_wichai | UPDATE | grades | grade_001 | 55 | 65 | SUCCESS |
| 2568-11-01 09:15:22 | student_somjai | SELECT | grades | all | - | - | SUCCESS |
| 2568-11-01 09:16:05 | student_somjai | UPDATE | grades | grade_002 | 70 | 95 | **FAILED** |
| 2568-11-01 10:00:00 | unknown | SELECT | teachers | all | - | - | **FAILED** |

> **สังเกต:** แถวที่ 3 — นักเรียนพยายามแก้เกรดแต่ล้มเหลว (Permission denied) และแถวที่ 4 — มีคนพยายามเข้าตาราง teachers แต่ถูกปฏิเสธ → ต้องสอบสวนทันที!

**บันทึก Audit Log ด้วย Python:**

```python
import sqlite3
from datetime import datetime

def log_action(db_path, username, action, table_name,
               record_id=None, old_value=None, new_value=None,
               ip_address='localhost', status='SUCCESS'):
    """บันทึกการกระทำลงใน Audit Log"""

    conn = sqlite3.connect(db_path)
    cursor = conn.cursor()

    cursor.execute("""
        INSERT INTO audit_log
            (username, action, table_name, record_id, old_value, new_value, ip_address, status)
        VALUES (?, ?, ?, ?, ?, ?, ?, ?)
    """, (username, action, table_name, record_id,
          str(old_value), str(new_value), ip_address, status))

    conn.commit()
    conn.close()

# ===== ตัวอย่างการใช้งาน =====

# ครูแก้เกรด
log_action(
    db_path='school_data.db',
    username='teacher_wichai',
    action='UPDATE',
    table_name='grades',
    record_id='grade_001',
    old_value={'score': 55},
    new_value={'score': 65},
    ip_address='192.168.1.15',
    status='SUCCESS'
)

# นักเรียนพยายามเข้าตารางที่ไม่มีสิทธิ์
log_action(
    db_path='school_data.db',
    username='student_somjai',
    action='SELECT',
    table_name='teachers',
    ip_address='192.168.1.22',
    status='FAILED - Permission Denied'
)
```

## 💉 SQL Injection — ภัยร้ายที่มาจากช่อง Input

**SQL Injection** (เอสคิวแอล อินเจคชัน) คือการโจมตีโดยการแทรกคำสั่ง SQL เข้าไปใน Input ของผู้ใช้ เพื่อหลอกให้ระบบรันคำสั่งที่ต้องการ

---

### 🔴 ตัวอย่างที่อันตราย — ช่อง Login ที่ไม่ปลอดภัย

สมมติมีระบบ Login แบบนี้:

```python
# ❌ โค้ดอันตราย! อย่าทำแบบนี้!

import sqlite3

def login_UNSAFE(username, password):
    conn = sqlite3.connect('school.db')
    cursor = conn.cursor()

    # สร้าง Query โดยนำ input ของผู้ใช้ใส่ตรง ๆ — อันตรายมาก!
    query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
    print(f"Query ที่รัน: {query}")

    cursor.execute(query)
    result = cursor.fetchone()
    conn.close()
    return result is not None
```

**การโจมตีแบบ SQL Injection:**

```
ผู้ใช้ทั่วไปพิมพ์:
   username: "admin"
   password: "wrongpass"
   → Query: SELECT * FROM users WHERE username='admin' AND password='wrongpass'
   → ผลลัพธ์: ไม่พบ → Login ล้มเหลว ✅ (ปกติ)

แฮกเกอร์พิมพ์:
   username: "admin' --"     ← มีอัฒภาค (') และ --
   password: "อะไรก็ได้"
   → Query: SELECT * FROM users WHERE username='admin' --' AND password='อะไรก็ได้'
   → SQL เห็นว่า: -- คือ comment → ตัดทุกอย่างหลังนั้นทิ้ง
   → Query จริง: SELECT * FROM users WHERE username='admin'
   → ผลลัพธ์: พบ admin → Login สำเร็จ!!! ❌ (อันตราย!)

การโจมตีขั้นสุด:
   username: "' OR '1'='1' --"
   password: "อะไรก็ได้"
   → Query: SELECT * FROM users WHERE username='' OR '1'='1' --' AND password='...'
   → '1'='1' เป็นจริงเสมอ → เข้าระบบได้โดยไม่ต้องรู้รหัสผ่าน!
```

**การโจมตีที่รุนแรงกว่า — DROP TABLE:**

```
username: "admin'; DROP TABLE students; --"
→ Query: SELECT * FROM users WHERE username='admin'; DROP TABLE students; --'
→ ลบตาราง students ทิ้งทั้งตาราง!!!! 💀
```

---

### ✅ วิธีป้องกัน — Parameterized Query (พาราเมทไรซด์ เควรี่)

**Parameterized Query** (หรือ Prepared Statement) คือการแยกข้อมูลกับคำสั่ง SQL ออกจากกัน ใช้ `?` เป็น placeholder แทน

```python
# ✅ โค้ดที่ปลอดภัย

import sqlite3

def login_SAFE(username, password):
    conn = sqlite3.connect('school.db')
    cursor = conn.cursor()

    # ใช้ ? แทนค่าจริง แล้วส่งค่าเป็น tuple แยก
    query = "SELECT * FROM users WHERE username = ? AND password = ?"
    cursor.execute(query, (username, password))  # ← ส่งค่าแยกจาก Query!

    result = cursor.fetchone()
    conn.close()
    return result is not None

# ทดสอบ
print(login_SAFE("admin", "correctpassword"))      # True (ถ้ารหัสถูก)
print(login_SAFE("admin' --", "อะไรก็ได้"))       # False (Injection ไม่ได้ผล!)
print(login_SAFE("' OR '1'='1' --", "pass"))       # False (ปลอดภัย!)
```

**ทำไม Parameterized Query ถึงปลอดภัย?**

```
แฮกเกอร์พิมพ์: "admin' --"

แบบไม่ปลอดภัย: ใส่ค่าตรงๆ ใน String
   Query: WHERE username='admin' --'   ← SQL อ่านแล้วรัน!

แบบปลอดภัย: ใช้ Parameterized Query
   ระบบมองว่า "admin' --" คือ string ธรรมดา ไม่ใช่ SQL
   Query: WHERE username=?  → ค่า ? = "admin' --" (ทั้งก้อน)
   ผลลัพธ์: ไม่พบ username ที่ชื่อ "admin' --" → Login ล้มเหลว ✅
```

### ตัวอย่างสมบูรณ์ — ระบบค้นหาที่ปลอดภัย

```python
import sqlite3

def search_accidents_SAFE(province: str, year: int):
    """
    ค้นหาอุบัติเหตุตามจังหวัดและปี — ปลอดภัยจาก SQL Injection
    """
    conn = sqlite3.connect('school_data.db')
    cursor = conn.cursor()

    # ✅ ใช้ ? สำหรับทุกค่าที่รับมาจาก User
    query = """
        SELECT accident_id, วันที่เกิดเหตุ, จังหวัด, ความรุนแรง
        FROM road_accidents
        WHERE จังหวัด = ?
          AND ปี_พ.ศ. = ?
        ORDER BY วันที่เกิดเหตุ
        LIMIT 10
    """

    cursor.execute(query, (province, year))  # ← ส่ง parameter แยก
    results = cursor.fetchall()
    conn.close()

    return results

# ใช้งาน
results = search_accidents_SAFE("เชียงใหม่", 2568)
for row in results:
    print(row)

# ถึงแฮกเกอร์จะใส่ injection ก็ไม่ได้ผล
safe_results = search_accidents_SAFE("'; DROP TABLE road_accidents; --", 2568)
# ได้รายการว่าง [] เพราะไม่มีจังหวัดชื่อนั้น ไม่ได้ DROP อะไร!
```

## 🛠️ Workshop 4.4 — สาธิต SQL Injection และป้องกัน

**โจทย์:**

**ส่วนที่ 1 — เข้ารหัส Password:**
1. เขียนฟังก์ชัน `hash_password(password)` ที่ใช้ SHA-256 + Salt
2. เขียนฟังก์ชัน `verify_password(password, salt, hash)` สำหรับ Login
3. ทดสอบ: ลองใช้รหัสถูก 1 ครั้ง และรหัสผิด 1 ครั้ง

**ส่วนที่ 2 — SQL Injection:**
1. เขียนฟังก์ชัน `login_unsafe(username, password)` แบบเสี่ยง
2. ทดสอบ Injection ด้วย `"admin' --"` และสังเกตผล
3. แก้เป็น `login_safe(username, password)` ด้วย Parameterized Query
4. ทดสอบ Injection อีกครั้ง → ต้องได้ False

**ส่วนที่ 3 — Audit Log:**
1. สร้างตาราง `audit_log` ในฐานข้อมูล
2. เรียก `log_action()` บันทึกทุกครั้งที่มีการ Login (ทั้งสำเร็จและล้มเหลว)
3. Query `audit_log` แสดง 10 รายการล่าสุด

## 📋 สรุปเทคโนโลยีความปลอดภัยที่เรียน

| เทคโนโลยี | คำอ่าน | ป้องกันอะไร | ใช้ตอนไหน |
|-----------|--------|-----------|----------|
| **Hash + Salt** | แฮช + ซอลท์ | Password รั่วไหล | เก็บรหัสผ่านในฐานข้อมูล |
| **GRANT/REVOKE** | แกรนท์/รีโวค | เข้าถึงโดยไม่มีสิทธิ์ | กำหนดสิทธิ์ผู้ใช้ |
| **Parameterized Query** | พาราเมทไรซด์ เควรี่ | SQL Injection | รับ Input จากผู้ใช้ |
| **3-2-1 Backup** | ทรี-ทู-วัน แบ็กอัป | ข้อมูลสูญหาย | ทำเป็นประจำทุกวัน |
| **Audit Log** | ออดิท ล็อก | การดัดแปลงข้อมูล | ทุก Action ในระบบ |
| **Encryption** | เอ็นคริปชัน | ข้อมูลถูกดักอ่าน | ส่งข้อมูลผ่านเครือข่าย |
| **Risk Matrix** | ริสก์ เมทริกซ์ | วางแผนป้องกันผิดจุด | ประเมินระบบก่อน Deploy |
