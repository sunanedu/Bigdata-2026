## 📊 ประเภทที่ 1 — Structured Data (สตรักเจอร์ด เดต้า) — "ข้อมูลมีโครงสร้าง"

**ความหมาย:** ข้อมูลที่ถูกจัดเรียงในรูปแบบตาราง (แถวและคอลัมน์) ที่ชัดเจน สามารถค้นหาและประมวลผลด้วย SQL ได้ทันที

> **เปรียบเทียบ:** เหมือนสมุดโทรศัพท์ที่มีคอลัมน์ ชื่อ | เบอร์ | ที่อยู่ ชัดเจนทุกช่อง

**ตัวอย่างในชีวิตจริง:**

| ตัวอย่าง | ลักษณะ |
|---------|--------|
| ตาราง Excel คะแนนนักเรียน | ชื่อ, วิชา, คะแนน — ทุกแถวมีรูปแบบเดียวกัน |
| ข้อมูลอุบัติเหตุ `thailand_road_accidents_2568.csv` | 51 คอลัมน์, 20,000 แถว — โครงสร้างตายตัว |
| ฐานข้อมูลธนาคาร | เลขบัญชี, ชื่อ, ยอดเงิน — แต่ละบัญชีมีฟิลด์เดียวกัน |
| ตารางพนักงาน HR | รหัสพนักงาน, ชื่อ, แผนก, เงินเดือน |

**รูปแบบไฟล์ที่พบบ่อย:**

```
✅ CSV (Comma-Separated Values) — ซีเอสวี
   ตัวอย่าง: students.csv
   accident_id,วันที่เกิดเหตุ,จังหวัด,ความรุนแรง
   ACC2568000001,2568-01-09,สมุทรสาคร,บาดเจ็บเล็กน้อย
   ACC2568000002,2568-01-10,เชียงใหม่,เสียชีวิต

✅ Excel (.xlsx)
   เหมือน CSV แต่มีหลาย Sheet และ formatting

✅ SQL Table ในฐานข้อมูล (MySQL, SQLite)
   CREATE TABLE students (id INT, name TEXT, score REAL)
```

**ข้อดีและข้อเสีย:**

| ข้อดี | ข้อเสีย |
|-------|---------|
| ค้นหาด้วย SQL ได้เลย | โครงสร้างตายตัว — เพิ่มคอลัมน์ลำบาก |
| วิเคราะห์ด้วย Pandas ง่าย | ไม่เหมาะกับข้อมูลที่แต่ละแถวมีฟิลด์ต่างกัน |
| เครื่องมือรองรับมาก | ขนาดไฟล์ใหญ่เมื่อมีข้อมูลซ้ำ |

## 📋 ประเภทที่ 2 — Semi-structured Data (เซมิ-สตรักเจอร์ด เดต้า) — "ข้อมูลกึ่งโครงสร้าง"

**ความหมาย:** ข้อมูลที่ **มีโครงสร้างบางส่วน** แต่ไม่ตายตัวเหมือนตาราง แต่ละรายการอาจมีฟิลด์ที่แตกต่างกัน ใช้ "แท็ก" หรือ "key" เพื่อบอกความหมายของข้อมูล

> **เปรียบเทียบ:** เหมือนพัสดุไปรษณีย์ — มีที่อยู่ผู้ส่งและผู้รับเสมอ แต่ของข้างในอาจต่างกันได้

**ตัวอย่างรูปแบบไฟล์:**

### JSON (เจสัน) — JavaScript Object Notation

**JSON** คือรูปแบบข้อมูลที่ใช้ key-value pair จัดเก็บข้อมูล อ่านง่ายทั้งคนและคอมพิวเตอร์ ใช้กันมากที่สุดในโลก API

```json
{
  "accident_id": "ACC2568000001",
  "วันที่": "2568-01-09",
  "จังหวัด": "สมุทรสาคร",
  "ผู้เกี่ยวข้อง": [
    {
      "บทบาท": "คนขับ",
      "อายุ": 32,
      "เพศ": "ชาย",
      "อาการบาดเจ็บ": "บาดเจ็บเล็กน้อย"
    },
    {
      "บทบาท": "ผู้โดยสาร",
      "อายุ": 25,
      "เพศ": "หญิง",
      "อาการบาดเจ็บ": "ไม่บาดเจ็บ"
    }
  ],
  "ยานพาหนะ": {
    "ประเภท": "รถจักรยานยนต์",
    "ยี่ห้อ": "Honda",
    "ปีที่ผลิต": 2021
  }
}
```

> **สังเกต:** ข้อมูล JSON ข้างต้น 1 อุบัติเหตุ อาจมีผู้เกี่ยวข้อง 1, 2, หรือ 10 คนก็ได้ — ใส่ใน CSV ไม่ได้!

### XML (เอ็กซ์เอ็มแอล) — eXtensible Markup Language

```xml
<accident>
  <accident_id>ACC2568000001</accident_id>
  <วันที่>2568-01-09</วันที่>
  <จังหวัด>สมุทรสาคร</จังหวัด>
  <ความรุนแรง>บาดเจ็บเล็กน้อย</ความรุนแรง>
</accident>
```

### YAML (ยาเมิล) — Yet Another Markup Language

```yaml
accident_id: ACC2568000001
วันที่: 2568-01-09
จังหวัด: สมุทรสาคร
ผู้เกี่ยวข้อง:
  - บทบาท: คนขับ
    อายุ: 32
  - บทบาท: ผู้โดยสาร
    อายุ: 25
```

**เปรียบเทียบ 3 รูปแบบ:**

| รูปแบบ | อ่านง่าย? | ขนาดไฟล์ | ใช้บ่อยที่สุดใน |
|--------|-----------|----------|----------------|
| JSON | ✅ ดี | เล็ก | Web API, MongoDB, JavaScript |
| XML | ⚠️ พอใช้ | ใหญ่กว่า | ระบบรัฐบาล, ธนาคาร เก่า |
| YAML | ✅ ดีมาก | เล็กสุด | Config Files, Docker, Kubernetes |

## 🗂️ ประเภทที่ 3 — Unstructured Data (อันสตรักเจอร์ด เดต้า) — "ข้อมูลไม่มีโครงสร้าง"

**ความหมาย:** ข้อมูลที่ **ไม่มีโครงสร้างตายตัว** ไม่มีตาราง ไม่มี key-value คอมพิวเตอร์ต้องใช้ AI/ML เพื่อเข้าใจความหมาย

> **เปรียบเทียบ:** เหมือนสมุดบันทึกส่วนตัว — เขียนอะไรก็ได้ ไม่มีรูปแบบ คนอื่นอ่านต้องตีความเอง

**ตัวอย่างข้อมูลไม่มีโครงสร้าง:**

```
📸 รูปภาพ (Images)
   - รูปถ่ายสถานที่เกิดอุบัติเหตุ
   - ภาพกล้องวงจรปิด
   - รูปภาพโซเชียลมีเดีย

🎵 เสียง (Audio)
   - บันทึกเสียงผู้แจ้งเหตุ 191
   - Podcast
   - เพลง MP3

🎥 วิดีโอ (Video)
   - กล้องติดรถ (Dashcam) บันทึกอุบัติเหตุ
   - YouTube
   - TikTok

📄 เอกสารข้อความอิสระ (Free Text)
   - รายงานตำรวจ: "รถจักรยานยนต์วิ่งมาด้วยความเร็ว
     แล้วพุ่งชนรถบรรทุกที่จอดอยู่ข้างทาง..."
   - รีวิวสินค้า: "สินค้าดีมาก ส่งเร็ว บรรจุภัณฑ์สวย"
   - ข่าวออนไลน์, บทความ, บล็อก

🗺️ PDF
   - แบบฟอร์มใบรายงานอุบัติเหตุ (กรอกมือ)
   - สัญญา, เอกสารกฎหมาย
```

**ทำไม Unstructured Data ถึงน่าสนใจ?**

```
ในโลก Big Data:
   80% ของข้อมูลทั้งหมด = Unstructured Data!
   20% ของข้อมูลทั้งหมด = Structured + Semi-structured

แต่ในอดีต เราใช้ได้แค่ 20% นั้น!
ปัจจุบัน AI/ML ช่วยให้เราเริ่มใช้ 80% ที่เหลือได้
```

## 🧩 โครงสร้าง JSON คืออะไร?

**JSON** (เจสัน) ย่อมาจาก **JavaScript Object Notation** (จาวาสคริปต์ ออบเจกต์ โนเทชัน) คือรูปแบบมาตรฐานสำหรับแลกเปลี่ยนข้อมูลบนอินเทอร์เน็ต

> **เปรียบเทียบ:** JSON เหมือน "ภาษากลาง" ที่โปรแกรมต่าง ๆ ใช้คุยกัน — เหมือนคนไทยและคนญี่ปุ่นคุยกันด้วยภาษาอังกฤษ

---

### องค์ประกอบพื้นฐานของ JSON

#### 1. Object (ออบเจกต์) — ใช้ `{ }` คู่ปีกกา

Object คือการเก็บข้อมูลแบบ **key: value** (กุญแจ: ค่า)

```json
{
  "ชื่อ": "สมชาย",
  "อายุ": 25,
  "เป็นนักเรียน": true
}
```

**กฎสำคัญ:**
- key ต้องเป็น String (ใส่เครื่องหมาย `""` เสมอ)
- value อาจเป็น String, Number, Boolean, Array, Object หรือ null
- คู่ key-value คั่นด้วย `,` (ยกเว้นรายการสุดท้าย)

#### 2. Array (อาร์เรย์) — ใช้ `[ ]` วงเล็บเหลี่ยม

Array คือรายการของข้อมูลหลายชิ้นเรียงกัน

```json
["กรุงเทพ", "เชียงใหม่", "ขอนแก่น", "สงขลา"]
```

หรือ Array ของ Object:

```json
[
  {"ชื่อ": "สมชาย", "คะแนน": 85},
  {"ชื่อ": "สมหญิง", "คะแนน": 92},
  {"ชื่อ": "สมศรี", "คะแนน": 78}
]
```

#### 3. Nested Object (เนสเตด ออบเจกต์) — Object ซ้อน Object

```json
{
  "accident_id": "ACC2568000001",
  "สถานที่": {
    "จังหวัด": "สมุทรสาคร",
    "อำเภอ": "สะพานสูง",
    "พิกัด": {
      "latitude": 10.894093,
      "longitude": 102.460892
    }
  },
  "ยานพาหนะ": [
    {
      "ลำดับ": 1,
      "ประเภท": "รถจักรยานยนต์",
      "ยี่ห้อ": "Honda",
      "ปี": 2021,
      "ความเสียหาย": "ชนหนัก"
    }
  ]
}
```

---

### ประเภทของ Value ใน JSON

| ประเภท | ตัวอย่าง | ใช้แทน |
|--------|---------|--------|
| String (สตริง) | `"สมชาย"`, `"2568-01-09"` | ข้อความ, วันที่ |
| Number (นัมเบอร์) | `25`, `98.6`, `-5` | ตัวเลขจำนวนเต็มหรือทศนิยม |
| Boolean (บูลีน) | `true`, `false` | ค่าจริง/เท็จ |
| Array (อาร์เรย์) | `[1, 2, 3]` | รายการ |
| Object (ออบเจกต์) | `{"key": "value"}` | ข้อมูลแบบกลุ่ม |
| null (นัล) | `null` | ค่าว่าง/ไม่มีข้อมูล |

## 🐍 อ่านและเขียน JSON ด้วย Python

Python มี library ชื่อ `json` มาให้ในตัว ไม่ต้อง pip install

### การอ่าน JSON จากไฟล์ — `json.load()`

```python
import json

# สมมติมีไฟล์ sample_api_data.json
# เก็บข้อมูลอุบัติเหตุในรูปแบบ JSON

with open('unit05_json/sample_api_data.json', 'r', encoding='utf-8') as f:
    data = json.load(f)  # ← อ่านไฟล์แล้วแปลงเป็น Python dict/list

# ดูโครงสร้าง
print(type(data))          # <class 'list'> หรือ <class 'dict'>
print(len(data))           # จำนวนรายการ

# ดูรายการแรก
print(data[0])
```

**ผลลัพธ์ที่ได้:**
```
<class 'list'>
5
{'accident_id': 'ACC2568000001', 'วันที่': '2568-01-09', 'จังหวัด': 'สมุทรสาคร', ...}
```

### การอ่าน JSON จาก String — `json.loads()`

บางครั้งเราได้ JSON มาเป็น String (เช่น จาก API Response)

```python
import json

# สมมติ response จาก API
json_string = '{"accident_id": "ACC001", "จังหวัด": "กรุงเทพ", "ผู้เสียชีวิต": 0}'

# แปลง String → Python dict
data = json.loads(json_string)   # ← loads = load string

print(data['จังหวัด'])    # กรุงเทพ
print(data['ผู้เสียชีวิต'])  # 0
```

### 🔧 แปลง Nested JSON ให้แบน (Flatten) ด้วย json_normalize()

## 🤔 ทำไมต้องมี NoSQL?

ย้อนกลับไปที่ตัวอย่างอุบัติเหตุ — ลองดูปัญหาถ้าเก็บข้อมูลแบบ SQL ตาราง:

```
ปัญหาที่ 1: ข้อมูลผู้เกี่ยวข้องไม่เท่ากัน

อุบัติเหตุ A มีผู้เกี่ยวข้อง 2 คน
อุบัติเหตุ B มีผู้เกี่ยวข้อง 7 คน
อุบัติเหตุ C มีผู้เกี่ยวข้อง 0 คน

ถ้าใช้ SQL ต้องสร้างตาราง persons แยก แล้ว JOIN
ถ้าใช้ MongoDB เก็บเป็น Array ในเอกสารเดียว!

ปัญหาที่ 2: โครงสร้างข้อมูลเปลี่ยนบ่อย

เดือนแรก: เก็บแค่ ชื่อ, อายุ, เพศ
เดือนที่สาม: ต้องเพิ่ม พิกัด GPS
เดือนที่หก: ต้องเพิ่ม รูปถ่ายสถานที่

SQL: ต้อง ALTER TABLE ทุกครั้ง — ยุ่งยาก
MongoDB: เพิ่ม field ได้เลยโดยไม่กระทบเอกสารเก่า!

ปัญหาที่ 3: ข้อมูลปริมาณมหาศาล ต้องกระจายหลายเซิร์ฟเวอร์

SQL: ขยายแนวตั้ง (เซิร์ฟเวอร์แรงขึ้น) — ราคาแพง
NoSQL: ขยายแนวนอน (เพิ่มเซิร์ฟเวอร์) — ง่ายกว่า
```

---

## 📚 แนวคิดหลัก: SQL vs NoSQL

| หัวข้อ | SQL (Relational) | NoSQL (MongoDB) |
|--------|-----------------|-----------------|
| โครงสร้างข้อมูล | **ตาราง (Table)** ← แถวและคอลัมน์ | **เอกสาร (Document)** ← JSON |
| หน่วยเก็บข้อมูล | Row (แถว) | Document (เอกสาร) |
| กลุ่มข้อมูล | Table (ตาราง) | Collection (คอลเลคชัน) |
| ภาษาค้นหา | SQL | MongoDB Query Language (MQL) |
| Schema | ตายตัว (Rigid) | ยืดหยุ่น (Flexible) |
| ความสัมพันธ์ | JOIN หลายตาราง | ฝังไว้ในเอกสารเดียว |
| เหมาะกับ | ข้อมูลที่มีโครงสร้างชัดเจน | ข้อมูลที่โครงสร้างเปลี่ยนได้ |

---

## 🗃️ MongoDB Document Model — โมเดลข้อมูลใน MongoDB

> **เปรียบเทียบ:** ถ้า SQL เหมือนสมุดตาราง Excel ที่ทุกแถวต้องมีคอลัมน์เดียวกัน MongoDB เหมือนลิ้นชักเก็บกระดาษ — แต่ละแผ่น (Document) จะมีข้อมูลต่างกันก็ได้

```
ระบบข้อมูลใน MongoDB:

Database (ดาต้าเบส)           ← เหมือน SQL Database
  └── Collection (คอลเลคชัน)  ← เหมือน SQL Table
        ├── Document (ดอกคิวเม้นต์)  ← เหมือน SQL Row
        │     ├── Field: "accident_id": "ACC001"
        │     ├── Field: "จังหวัด": "กรุงเทพ"
        │     └── Field: "ยานพาหนะ": [...]
        ├── Document
        └── Document
```

**ตัวอย่าง Document ใน MongoDB:**

```json

      "ประเภท": "รถจักรยานยนต์",

    }

  "ผู้เกี่ยวข้อง": [
    {
      "บทบาท": "คนขับ",
      "อายุ": 32,
```

```python
# เลือกเฉพาะบางฟิลด์ (Projection)
# 1 = แสดง, 0 = ซ่อน
selected = accidents.find(

    {"accident_id": 1, "จังหวัด": 1, "ความรุนแรง": 1, "_id": 0}
```

### 3. Query แบบมีเงื่อนไข — Comparison Operators

MongoDB ใช้ **Operator** พิเศษในการกรองข้อมูล:

| Operator | คำอ่าน | ความหมาย | เทียบกับ SQL |
|----------|--------|---------|------------|
| `$eq` | ดอลลาร์-อีคิว | เท่ากับ | `= value` |
| `$ne` | ดอลลาร์-เอ็นอี | ไม่เท่ากับ | `!= value` |
| `$gt` | ดอลลาร์-จีที | มากกว่า | `> value` |
| `$gte` | ดอลลาร์-จีทีอี | มากกว่าหรือเท่ากับ | `>= value` |
| `$lt` | ดอลลาร์-แอลที | น้อยกว่า | `< value` |
| `$lte` | ดอลลาร์-แอลทีอี | น้อยกว่าหรือเท่ากับ | `<= value` |
| `$in` | ดอลลาร์-อิน | อยู่ในรายการ | `IN (...)` |
| `$and` | ดอลลาร์-แอนด์ | และ | `AND` |
| `$or` | ดอลลาร์-ออร์ | หรือ | `OR` |

**ตัวอย่างการใช้งาน:**

```python
# หาอุบัติเหตุที่มีผู้เสียชีวิตมากกว่า 0
fatal = accidents.find({"ผู้เสียชีวิต": {"$gt": 0}})

# เทียบกับ SQL: SELECT * FROM accidents WHERE ผู้เสียชีวิต > 0

# หาผู้บาดเจ็บระหว่าง 2-5 คน
moderate = accidents.find({
    "ผู้บาดเจ็บ": {"$gte": 2, "$lte": 5}
})
# SQL: WHERE ผู้บาดเจ็บ BETWEEN 2 AND 5

# หาอุบัติเหตุในภาคเหนือและภาคอีสาน
north_east = accidents.find({
    "จังหวัด": {"$in": ["เชียงใหม่", "เชียงราย", "ขอนแก่น", "อุดรธานี"]}
})
# SQL: WHERE จังหวัด IN ('เชียงใหม่', 'เชียงราย', 'ขอนแก่น', 'อุดรธานี')

# หาอุบัติเหตุรถจักรยานยนต์ที่เสียชีวิต ($and)
moto_fatal = accidents.find({
    "$and": [
        {"ยานพาหนะ": "รถจักรยานยนต์"},
        {"ผู้เสียชีวิต": {"$gt": 0}}
    ]
})
# SQL: WHERE ยานพาหนะ = 'รถจักรยานยนต์' AND ผู้เสียชีวิต > 0

# นับจำนวน
count = accidents.count_documents({"ความรุนแรง": "เสียชีวิต"})
print(f"อุบัติเหตุเสียชีวิต: {count} ครั้ง")
```

### 4. updateOne (อัปเดท วัน) — แก้ไขเอกสาร

```python
# ถ้าไม่มี field นั้นอยู่ → สร้างใหม่

# เพิ่ม field ใหม่ให้เอกสาร
accidents.update_one(
```

## 🤔 เมื่อไหร่ใช้อะไร? — Decision Framework

> **Decision Framework** (ดีซิชัน เฟรมเวิร์ก) = กรอบการตัดสินใจ — ชุดคำถามที่ช่วยให้เลือกเครื่องมือได้ถูกต้อง

```
คำถามที่ 1: ข้อมูลมีโครงสร้างตายตัวไหม?

   ✅ ใช่ (ทุกแถวมีคอลัมน์เดียวกัน)
      → ใช้ SQL (SQLite / MySQL)
      → ตัวอย่าง: ข้อมูลอุบัติเหตุ, คะแนนนักเรียน, ยอดขาย

   ❌ ไม่ (แต่ละรายการมี field ต่างกัน หรือมี nested object)
      → ไปคำถามที่ 2

คำถามที่ 2: ต้องการ Query ซับซ้อน (JOIN หลายตาราง, Aggregate) ไหม?

   ✅ ใช่
      → ใช้ SQL ต่อไป แม้จะต้อง Normalize ข้อมูล
      → ออกแบบตาราง แล้วใช้ JOIN

   ❌ ไม่ (Query ง่าย แต่ข้อมูลมีโครงสร้างยืดหยุ่น)
      → ใช้ MongoDB (NoSQL)
      → เก็บเป็น Document เพื่อความยืดหยุ่น

คำถามที่ 3: ต้องการประมวลผลหรือวิเคราะห์ข้อมูลในหน่วยความจำไหม?

   ✅ ใช่ (คำนวณ, ทำ visualization, Machine Learning)
      → ใช้ Python + Pandas
      → โหลดข้อมูลจาก SQL/MongoDB → DataFrame → วิเคราะห์

   ❌ ไม่ (แค่เก็บ/ดึงข้อมูล)
      → ใช้ SQL หรือ MongoDB อย่างใดอย่างหนึ่ง
```

## 🚰 Simple Data Pipeline — ท่อข้อมูลอย่างง่าย

**Data Pipeline** (เดต้า ไปป์ไลน์) = เส้นทางที่ข้อมูลเดินทางจากต้นทางถึงปลายทาง เหมือนท่อน้ำ — น้ำไหลจากต้นน้ำผ่านกระบวนการต่างๆ จนถึงก๊อกน้ำ

```
Pipeline ของหลักสูตรนี้:

แหล่งข้อมูล          ประมวลผล              จัดเก็บ           นำเสนอ
   │                     │                    │                  │
   ▼                     ▼                    ▼                  ▼
CSV ไฟล์    →  Python + Pandas   →   SQLite DB    →  Dashboard HTML
JSON API    →  (ทำความสะอาด)     →   MongoDB      →  (Chart.js)
                (แปลงรูปแบบ)
```

**ตัวอย่าง Pipeline เตรียมข้อมูลสำหรับ Dashboard:**

```python
# ===== Simple Data Pipeline =====
# ขั้นตอน: CSV → Python → ทำความสะอาด → JSON → Dashboard

import pandas as pd
import json
import sqlite3

print("=" * 50)
print("DATA PIPELINE: อุบัติเหตุ → Dashboard JSON")
print("=" * 50)

# ===== STEP 1: EXTRACT (สกัดข้อมูล) =====
print("\n[1] กำลังโหลดข้อมูล CSV...")
df = pd.read_csv('thailand_road_accidents_2568.csv', encoding='utf-8-sig')
print(f"    โหลดสำเร็จ: {df.shape[0]:,} แถว, {df.shape[1]} คอลัมน์")

# ===== STEP 2: TRANSFORM (แปลงข้อมูล) =====
print("\n[2] กำลังประมวลผลข้อมูล...")

# 2.1 สรุปอุบัติเหตุตามจังหวัด (Top 10)
province_summary = (
    df.groupby('จังหวัด')
    .agg(
        จำนวนครั้ง=('accident_id', 'count'),
        ผู้เสียชีวิต=('จำนวนผู้เสียชีวิต', 'sum'),
        ผู้บาดเจ็บ=('จำนวนผู้บาดเจ็บรวม', 'sum')
    )
    .sort_values('จำนวนครั้ง', ascending=False)
    .head(10)
    .reset_index()
)
print(f"    Top 10 จังหวัด พร้อมแล้ว")

# 2.2 สรุปตามยานพาหนะ
vehicle_summary = (
    df.groupby('ยานพาหนะหลัก')
    .agg(จำนวน=('accident_id', 'count'))
    .sort_values('จำนวน', ascending=False)
    .reset_index()
)
print(f"    สรุปยานพาหนะ พร้อมแล้ว")

# 2.3 สรุปรายเดือน
monthly_summary = (
    df.groupby('เดือน')
    .agg(
        จำนวนครั้ง=('accident_id', 'count'),
        ผู้เสียชีวิต=('จำนวนผู้เสียชีวิต', 'sum')
    )
    .reset_index()
)
print(f"    สรุปรายเดือน พร้อมแล้ว")

# ===== STEP 3: LOAD (โหลดลงปลายทาง) =====
print("\n[3] กำลัง Export เป็น JSON...")

dashboard_data = {
    "สรุปรวม": {
        "อุบัติเหตุทั้งหมด": int(df.shape[0]),
        "ผู้เสียชีวิตรวม": int(df['จำนวนผู้เสียชีวิต'].sum()),
        "ผู้บาดเจ็บรวม": int(df['จำนวนผู้บาดเจ็บรวม'].sum()),
    },
    "top10_จังหวัด": province_summary.to_dict('records'),
    "ยานพาหนะ": vehicle_summary.to_dict('records'),
    "รายเดือน": monthly_summary.to_dict('records')
}

with open('dashboard_data.json', 'w', encoding='utf-8') as f:
    json.dump(dashboard_data, f, ensure_ascii=False, indent=2)

print("    ✅ บันทึก dashboard_data.json สำเร็จ!")
print("\n[4] Pipeline เสร็จสมบูรณ์!")
print("    ไฟล์ dashboard_data.json พร้อมใช้ใน Dashboard HTML แล้ว")
```

## 🎯 สรุปเปรียบเทียบ SQL vs MongoDB แบบเคียงข้างกัน

```
งาน: ค้นหาอุบัติเหตุที่มีผู้เสียชีวิต > 0 ในจังหวัดเชียงใหม่

─────── SQL (SQLite) ───────────────────────────────
SELECT accident_id, วันที่เกิดเหตุ, จังหวัด, จำนวนผู้เสียชีวิต
FROM road_accidents
WHERE จังหวัด = 'เชียงใหม่'
  AND จำนวนผู้เสียชีวิต > 0
ORDER BY วันที่เกิดเหตุ;

─────── MongoDB (Python) ───────────────────────────
results = accidents.find(
    {"$and": [
        {"จังหวัด": "เชียงใหม่"},
        {"จำนวนผู้เสียชีวิต": {"$gt": 0}}
    ]},
    {"accident_id": 1, "วันที่": 1, "จังหวัด": 1, "จำนวนผู้เสียชีวิต": 1, "_id": 0}
).sort("วันที่", 1)  # 1 = ASC

─────── ผลลัพธ์เหมือนกัน! ───────────────────────
แต่ MongoDB เหมาะเมื่อข้อมูลยืดหยุ่นและ schema เปลี่ยนได้
SQL เหมาะเมื่อโครงสร้างตายตัวและต้องการ JOIN
```

```
หน่วยที่ 5 — ข้อมูลไม่มีโครงสร้างและการจัดการข้อมูลหลากหลายรูปแบบ

📊 3 ประเภทข้อมูล:
   Structured    → ตาราง CSV, SQL, Excel — โครงสร้างตายตัว
   Semi-structured → JSON, XML, YAML — มี key-value แต่ยืดหยุ่น
   Unstructured  → รูปภาพ, วิดีโอ, ข้อความอิสระ — ไม่มีโครงสร้าง

📄 JSON:
   Object  = { } พวง key: value
   Array   = [ ] รายการ
   Nested  = Object ซ้อน Object
   json.load()   → อ่านจากไฟล์
   json.loads()  → อ่านจาก String
   json.dump()   → เขียนลงไฟล์

🔄 Pipeline: JSON → DataFrame → SQLite
   pd.DataFrame(data)              → แปลง list of dict
   df.to_sql('table', conn)        → บันทึกลง SQLite
   json_normalize(nested_data)     → แบน Nested JSON

🍃 MongoDB:
   Collection = กลุ่มเอกสาร (≈ SQL Table)
   Document   = เอกสาร JSON 1 ชิ้น (≈ SQL Row)
   insert_one() / insert_many()   → เพิ่ม
   find() / find_one()            → ค้นหา
   update_one() / update_many()   → แก้ไข
   delete_one() / delete_many()   → ลบ
   Operators: $eq $gt $lt $in $and $or

🔧 Decision Framework:
   โครงสร้างตายตัว + JOIN ซับซ้อน → SQL
   โครงสร้างยืดหยุ่น + ข้อมูลซ้อน → MongoDB
   วิเคราะห์/คำนวณ/Visualization   → Python + Pandas
   ส่งให้ Dashboard                → JSON
```
