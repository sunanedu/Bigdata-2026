# บทที่ 1.1 — Big Data คืออะไร?
**(2 ชั่วโมง)**

---

## 🔍 นิยามของ Big Data

**Big Data** (บิ๊ก เดต้า) หมายถึง ข้อมูลที่มีปริมาณมากจนเครื่องมือทั่วไปอย่าง Excel ไม่สามารถจัดการได้อย่างมีประสิทธิภาพ ทั้งในแง่ของขนาด ความเร็วในการเกิดขึ้น และความหลากหลายของรูปแบบ

> **เปรียบเทียบให้เห็นภาพ:**  
> ลองนึกถึงน้ำในแก้ว (ข้อมูลธรรมดา) กับน้ำในเขื่อน (Big Data)  
> เราใช้แก้วตักน้ำในแก้วได้ → แต่ต้องใช้ระบบท่อ + เครื่องสูบน้ำ จัดการน้ำในเขื่อน  
> เครื่องมือก็ต้องเปลี่ยนตามขนาดของข้อมูลเช่นกัน

---

## 🔢 หลักการ 5V ของ Big Data

5V คือ คุณลักษณะ 5 ประการที่ทำให้ข้อมูลเรียกว่า "ใหญ่" อย่างแท้จริง

---

### V1 — Volume (วอลิ่วม) — "ปริมาณ"

**ความหมาย:** ข้อมูลมีจำนวนมหาศาล ตั้งแต่ Terabyte จนถึง Petabyte หรือมากกว่า

| หน่วย | ขนาด | เปรียบเทียบ |
|-------|------|-------------|
| 1 Kilobyte (KB) | 1,000 bytes | จดหมาย 1 ฉบับ |
| 1 Megabyte (MB) | 1,000 KB | ภาพถ่าย 1 รูป |
| 1 Gigabyte (GB) | 1,000 MB | หนัง HD 1 เรื่อง |
| 1 Terabyte (TB) | 1,000 GB | หนัง HD 1,000 เรื่อง |
| 1 Petabyte (PB) | 1,000 TB | Gmail เก็บข้อมูลผู้ใช้งานทั้งหมดประมาณ ~10 PB |

**ตัวอย่างชีวิตจริง:**

> 🏥 **โรงพยาบาลศิริราช** มีผู้ป่วยเข้ารับบริการกว่า 3,000 คนต่อวัน  
> ข้อมูลผู้ป่วย 1 คน ประกอบด้วย ประวัติการรักษา, ผล Lab, ภาพ X-ray (ไฟล์ละ ~50 MB), ยาที่ได้รับ  
> ใน 1 ปี ข้อมูลสะสมมากกว่า **100 Terabyte** — Excel รับไม่ไหวแน่นอน!

---

### V2 — Velocity (เวลอซิตี้) — "ความเร็ว"

**ความหมาย:** ข้อมูลเกิดขึ้นและเปลี่ยนแปลงอย่างรวดเร็ว ต้องการการประมวลผลแบบ Real-time

**ตัวอย่างชีวิตจริง:**

> 💳 **ธนาคารกสิกรไทย** มีการทำธุรกรรมผ่าน K PLUS กว่า **5 ล้านครั้งต่อวัน**  
> = ประมาณ **58 ธุรกรรมต่อวินาที!**  
> ระบบต้องตรวจสอบบัตรเครดิตปลอม (Fraud) แบบ Real-time ภายใน **0.1 วินาที**  
> ถ้าช้ากว่านี้ → ผู้ร้ายโอนเงินออกไปแล้ว

---

### V3 — Variety (วาไรตี้) — "ความหลากหลาย"

**ความหมาย:** ข้อมูลมาในหลายรูปแบบ ทั้งที่มีโครงสร้างชัดเจนและไม่มีโครงสร้าง

```
ประเภทข้อมูล:
┌─────────────────────┐   ┌──────────────────────┐   ┌────────────────────────┐
│  Structured Data    │   │ Semi-Structured Data  │   │  Unstructured Data     │
│  (มีโครงสร้าง)     │   │  (กึ่งมีโครงสร้าง)  │   │  (ไม่มีโครงสร้าง)     │
├─────────────────────┤   ├──────────────────────┤   ├────────────────────────┤
│ • ตาราง Excel       │   │ • JSON               │   │ • รูปภาพ               │
│ • ฐานข้อมูล SQL     │   │ • XML                │   │ • วิดีโอ               │
│ • CSV               │   │ • อีเมล              │   │ • เสียง                │
└─────────────────────┘   └──────────────────────┘   │ • ข้อความ Social Media │
                                                       │ • PDF เอกสาร           │
                                                       └────────────────────────┘
```

**ตัวอย่างชีวิตจริง:**

> 🏪 **เซ็นทรัล ช็อปปิ้ง** เก็บข้อมูลลูกค้าจากหลายแหล่งพร้อมกัน:
> - บิลซื้อสินค้า (ตาราง SQL → Structured)
> - รีวิวสินค้าบน Facebook (ข้อความ → Unstructured)
> - ประวัติการสแกน QR Code (JSON → Semi-Structured)
> - ภาพกล้องวงจรปิดในห้าง (วิดีโอ → Unstructured)  
> ทีม Data ต้องนำข้อมูลทั้ง 4 ประเภทมาวิเคราะห์ร่วมกัน

---

### V4 — Veracity (เวอแรสซิตี้) — "ความน่าเชื่อถือ"

**ความหมาย:** ข้อมูลไม่ได้ถูกทุกครั้ง — มีข้อมูลที่ผิด, ขาดหาย, หรือขัดแย้งกันอยู่เสมอ

**ตัวอย่างชีวิตจริง:**

> 🚗 **ชุดข้อมูลอุบัติเหตุทางถนนประเทศไทย ปี 2568** ที่เราใช้เรียนในวิชานี้:
> - บางรายการมีค่าว่าง (NULL) ในช่องหมายเลขถนน เพราะเป็นถนนชนบทที่ไม่มีหมายเลข
> - อายุของผู้ขับขี่บางรายบันทึกเป็น 0 ปี — ชัดเจนว่าข้อมูลผิด
> - บางครั้งบันทึกจังหวัดเป็น "กรุงเทพ" บางครั้ง "กรุงเทพมหานคร" — ข้อมูลไม่สอดคล้อง  
>
> นักวิเคราะห์ข้อมูลต้องตรวจสอบและทำความสะอาดข้อมูลก่อนเสมอ (จะเรียนในหน่วยที่ 3)

---

### V5 — Value (แวลิ่ว) — "คุณค่า"

**ความหมาย:** ข้อมูลมีมากแต่ไม่ใช่ทุกชิ้นมีประโยชน์ — ต้องวิเคราะห์ให้ได้ข้อมูลเชิงลึกที่นำไปใช้ตัดสินใจได้จริง

**ตัวอย่างชีวิตจริง:**

> 📊 **กรมทางหลวง** มีข้อมูลอุบัติเหตุทั้งประเทศ แต่ข้อมูลดิบ ๆ ไม่มีประโยชน์  
> Value คือการนำข้อมูลมาวิเคราะห์จนได้คำตอบว่า:
> - **"โค้งหมายเลข 327 กม. 45 เกิดอุบัติเหตุมากกว่าจุดอื่น 3 เท่า → ต้องติดป้ายเตือนและไฟ"**
> - **"อุบัติเหตุวันเสาร์ช่วง 22:00-02:00 เกิดจากดื่มแล้วขับ 68% → ต้องตั้งด่านตรวจ"**  
>
> นี่คือ Value ที่แท้จริงของ Big Data — **เปลี่ยนข้อมูลให้กลายเป็นการตัดสินใจที่ช่วยชีวิตคนได้**

---

```
        │  V4 Veracity  → ต้องตรวจสอบ          │

        └─────────────────────────────────────┘
```

## 🔄 วงจรชีวิตของข้อมูล (Data Lifecycle)

**Data Lifecycle** (เดต้า ไลฟ์ไซเคิล) คือ เส้นทางที่ข้อมูลเดินทางตั้งแต่เก็บรวบรวมจนถึงนำไปใช้ประโยชน์

```
  ① เก็บรวบรวม   →   ② ประมวลผล   →   ③ วิเคราะห์   →   ④ นำเสนอ   →   ⑤ ตัดสินใจ
  (Collection)       (Processing)      (Analysis)        (Presentation)  (Decision)

  ตัวอย่าง (ข้อมูลอุบัติเหตุถนน):
  ① เจ้าหน้าที่ตำรวจกรอกข้อมูลอุบัติเหตุที่เกิด
  ② ระบบรวบรวมข้อมูลทั่วประเทศเข้าสู่ฐานข้อมูลกลาง
  ③ นักวิเคราะห์หาจุดอันตรายและสาเหตุหลัก
  ④ สร้าง Dashboard แสดงแผนที่จุดเสี่ยงทั่วประเทศ
  ⑤ กรมทางหลวงตัดสินใจติดตั้งไฟสัญญาณ/ป้ายเตือน
```

| **กรมขนส่งทางบก** | อุบัติเหตุทุกครั้งทั่วประเทศ | หาจุดอันตรายและแก้ไขถนน |

## 📈 แนวโน้ม Big Data ในปี 2026

- **AI ผสาน Big Data:** ใช้ Machine Learning วิเคราะห์ข้อมูลอัตโนมัติ เช่น คาดเดาอุบัติเหตุก่อนเกิด
- **Edge Computing (เอดจ์ คอมพิวติ้ง):** ประมวลผลข้อมูลที่อุปกรณ์เลย ไม่ต้องส่งไปเซิร์ฟเวอร์ เช่น กล้องจับความเร็วรถที่วิเคราะห์ได้ทันที
- **Real-time Dashboard:** ผู้บริหารดูข้อมูล Live บนมือถือได้ทุกที่ทุกเวลา
- **Data Privacy (เดต้า ไพรเวซี่):** กฎหมาย PDPA ไทยบังคับใช้เต็มรูปแบบ — ข้อมูลส่วนตัวต้องได้รับความยินยอมก่อนใช้

# บทที่ 1.2 — ระบบฐานข้อมูล (Database Systems)
**(4 ชั่วโมง)**

---

## 🗄️ ฐานข้อมูลคืออะไร?

**Database** (เดต้าเบส) หรือ **ฐานข้อมูล** คือ ที่เก็บข้อมูลที่จัดระเบียบไว้อย่างเป็นระบบ ค้นหาและแก้ไขได้สะดวก

> **เปรียบเทียบ:**  
> ไฟล์ Excel = ลิ้นชักใส่เอกสาร — ค้นหาเองด้วยมือ  
> ฐานข้อมูล = ระบบห้องสมุดที่มีบัตรสืบค้น — ค้นได้ทันทีด้วยคำสั่ง

---

## ⚡ SQL vs NoSQL — ความแตกต่างหลัก

### Relational Database (SQL) — ฐานข้อมูลเชิงสัมพันธ์

**SQL** (เอส-คิว-แอล) คือระบบฐานข้อมูลที่เก็บข้อมูลในรูปแบบ **ตาราง** (Table) คล้าย Excel แต่มีประสิทธิภาพสูงกว่ามาก

**โครงสร้างของ SQL Database:**
```
ฐานข้อมูล (Database)
└── ตาราง (Table) ← เหมือน Sheet ใน Excel
    ├── แถว (Row / Record) ← ข้อมูล 1 รายการ
    └── คอลัมน์ (Column / Field) ← หัวข้อข้อมูล
```

**ตัวอย่าง — ตารางอุบัติเหตุ (ข้อมูลจริงจาก thailand_road_accidents_2568.csv):**

| accident_id | วันที่เกิดเหตุ | จังหวัด | ประเภทยานพาหนะหลัก | ความรุนแรง | จำนวนผู้เสียชีวิต |
|-------------|--------------|---------|-------------------|------------|-------------------|
| ACC2568000001 | 2568-01-09 | สมุทรสาคร | รถยนต์ส่วนบุคคล | ไม่บาดเจ็บ | 0 |
| ACC2568000002 | 2568-07-30 | ชลบุรี | รถปิกอัพ/กระบะ | บาดเจ็บเล็กน้อย | 0 |
| ACC2568000003 | 2568-03-07 | มุกดาหาร | รถจักรยานยนต์ | ไม่บาดเจ็บ | 0 |

> นี่คือตัวอย่าง **Structured Data** ที่เหมาะกับ SQL — มีคอลัมน์ชัดเจน ทุกแถวมีโครงสร้างเหมือนกัน

**ตัวอย่าง SQL ที่ใช้กับข้อมูลอุบัติเหตุไทย:**
```sql
-- ถามว่า: จังหวัดไหนเกิดอุบัติเหตุมากที่สุด Top 5?
SELECT จังหวัด, COUNT(*) AS จำนวนเหตุ
FROM road_accidents
GROUP BY จังหวัด
ORDER BY จำนวนเหตุ DESC
LIMIT 5;
```

---

### Non-Relational Database (NoSQL) — ฐานข้อมูลแบบไม่มีโครงสร้างตาราง

**NoSQL** (โน-เอส-คิว-แอล) คือระบบฐานข้อมูลที่**ไม่ได้เก็บในรูปตาราง** เหมาะกับข้อมูลที่โครงสร้างไม่แน่นอน

**ตัวอย่าง NoSQL (MongoDB) — เก็บข้อมูลรีวิวอุบัติเหตุ:**
```json
{
  "accident_id": "ACC2568000001",
  "จังหวัด": "สมุทรสาคร",
  "รายละเอียด": "รถชนกันที่ทางแยก ไม่มีผู้บาดเจ็บ",
  "ภาพประกอบ": ["photo1.jpg", "photo2.jpg"],
  "พยาน": [
    {"ชื่อ": "สมชาย", "เบอร์": "080-xxx-xxxx"},
    {"ชื่อ": "สมหญิง", "เบอร์": "090-xxx-xxxx"}
  ],
  "ตำแหน่ง_GPS": {"lat": 10.894093, "lng": 102.460892}
}
```
> สังเกตว่า "พยาน" มี 2 คนในเหตุการณ์นี้ แต่เหตุการณ์อื่นอาจมี 5 คนหรือ 0 คน — ตาราง SQL รับไม่ได้ แต่ NoSQL รับได้

---

## 📊 เปรียบเทียบ SQL vs NoSQL

| ประเด็น | SQL (เช่น SQLite, MySQL) | NoSQL (เช่น MongoDB) |
|--------|------------------------|---------------------|
| โครงสร้างข้อมูล | ตาราง (Table) | Document, Key-Value, Graph |
| ความยืดหยุ่น | ต่ำ — ต้องกำหนดคอลัมน์ล่วงหน้า | สูง — แต่ละ record ต่างกันได้ |
| ภาษาที่ใช้ | SQL (มาตรฐานเดียวกัน) | แตกต่างกันตามแต่ละระบบ |
| เหมาะกับ | ข้อมูลมีโครงสร้างชัดเจน เช่น บัญชี, นักเรียน, อุบัติเหตุ | ข้อมูลที่โครงสร้างไม่แน่นอน เช่น โพสต์ Social Media, Log ระบบ |
| ตัวอย่าง | MySQL, SQLite, PostgreSQL | MongoDB, Redis, Cassandra |

---

## 🛠️ ฐานข้อมูลที่ใช้ในหลักสูตรนี้

### SQLite (เอส-คิว-ไลท์)

- ฐานข้อมูล SQL ที่เบาที่สุด — ทำงานเป็นแค่ไฟล์ `.db` ไฟล์เดียว
- ไม่ต้องการเซิร์ฟเวอร์ — เปิด-ปิดได้ทันที
- เหมาะสำหรับการเรียนรู้และโปรเจกต์ขนาดเล็ก-กลาง
- ใช้ร่วมกับ **DB Browser for SQLite** ที่มีหน้าต่าง GUI ใช้งานง่าย


## 📐 Entity Relationship Diagram (ERD)

**ERD** (อี-อาร์-ดี) ย่อมาจาก **Entity-Relationship Diagram** (เอนทิตี้-รีเลชันชิพ-ไดอะแกรม) คือ **แผนภาพที่แสดงความสัมพันธ์ระหว่างข้อมูล** — ใช้วางแผนก่อนสร้างฐานข้อมูลจริง เหมือนวาดแบบบ้านก่อนก่อสร้าง

**องค์ประกอบของ ERD:**

| สัญลักษณ์ | ชื่อ | ความหมาย | ตัวอย่าง |
|----------|------|----------|---------|
| สี่เหลี่ยม | Entity (เอนทิตี้) | สิ่งที่เราต้องการเก็บข้อมูล | นักเรียน, วิชา, ครู |
| วงรี | Attribute (แอททริบิ้วท์) | คุณลักษณะของ Entity | ชื่อ, อายุ, คะแนน |
| เส้นเชื่อม | Relationship (รีเลชันชิพ) | ความสัมพันธ์ระหว่าง Entity | นักเรียน "เรียน" วิชา |

**ตัวอย่าง ERD ระบบโรงเรียน:**

```
┌──────────────┐         ┌──────────────┐         ┌──────────────┐
│   students   │         │   enrollments│         │   subjects   │
│  (นักเรียน)  │         │  (การลงทะเบียน)│        │   (วิชา)     │
├──────────────┤         ├──────────────┤         ├──────────────┤
│ student_id 🔑│─────────│ student_id 🔗│         │ subject_id 🔑│
│ name         │         │ subject_id 🔗│─────────│ subject_name │
│ grade        │         │ score        │         │ credits      │
│ birthdate    │         │ semester     │         │ teacher_id 🔗│
└──────────────┘         └──────────────┘         └──────────────┘
```
> 🔑 = Primary Key | 🔗 = Foreign Key

---

## 🗝️ Primary Key และ Foreign Key

### Primary Key (ไพรมารี่ คีย์) — "กุญแจหลัก"

**ความหมาย:** คอลัมน์ที่ใช้**ระบุตัวตนของแต่ละแถวได้อย่างไม่ซ้ำกัน**

**กฎของ Primary Key:**
- ต้องไม่ซ้ำกัน (Unique)
- ต้องไม่ว่าง (Not Null)
- แต่ละตารางมีได้เพียง 1 Primary Key

**ตัวอย่าง:**

```
ตาราง road_accidents (อุบัติเหตุถนน):
┌──────────────────┬─────────────────┬───────────┐
│   accident_id 🔑  │ วันที่เกิดเหตุ   │ จังหวัด   │
├──────────────────┼─────────────────┼───────────┤
│ ACC2568000001    │ 2568-01-09      │ สมุทรสาคร │  ← แถวนี้ระบุด้วย ACC2568000001
│ ACC2568000002    │ 2568-07-30      │ ชลบุรี    │  ← แถวนี้ระบุด้วย ACC2568000002
│ ACC2568000003    │ 2568-03-07      │ มุกดาหาร  │  ← แถวนี้ระบุด้วย ACC2568000003
└──────────────────┴─────────────────┴───────────┘
  ↑ Primary Key — ไม่มีรหัสซ้ำกันเลยแม้แต่รายการเดียว
```

> **เปรียบเทียบ:** Primary Key เหมือนเลขบัตรประชาชน — คนไทยทุกคนมีเลข 13 หลักไม่ซ้ำกัน

---

### Foreign Key (ฟอเร็น คีย์) — "กุญแจนอก"

**ความหมาย:** คอลัมน์ที่**อ้างอิงไปยัง Primary Key ของตารางอื่น** — ใช้เชื่อมตารางสองตารางเข้าหากัน

**ตัวอย่าง — ระบบโรงเรียน:**

```sql
-- ตาราง students
student_id 🔑 | name  | grade
-------------|-------|------
STU001       | สมชาย | 10
STU002       | สมหญิง | 11

-- ตาราง enrollments
enroll_id 🔑 | student_id 🔗 | subject_id | score
------------|--------------|------------|------
ENR001      | STU001       | MAT101     | 85
ENR002      | STU001       | ENG101     | 72
ENR003      | STU002       | MAT101     | 91
```
> `student_id` ในตาราง enrollments เป็น Foreign Key ที่อ้างอิงไปยัง `student_id` ในตาราง students

---

## 🔗 ประเภทของความสัมพันธ์ (Relationships)

### 1:1 (One-to-One) — หนึ่งต่อหนึ่ง

**ความหมาย:** 1 รายการในตาราง A สัมพันธ์กับ **ไม่เกิน 1 รายการ** ในตาราง B

**ตัวอย่าง:**
```
นักเรียน 1 คน  ←→  บัตรนักเรียน 1 ใบ
(สมชาย)              (รหัส STU001)
```

---

### 1:N (One-to-Many) — หนึ่งต่อหลาย (**พบบ่อยที่สุด**)

**ความหมาย:** 1 รายการในตาราง A สัมพันธ์กับ **หลายรายการ** ในตาราง B

**ตัวอย่าง:**
```
ครู 1 คน  →  ดูแลนักเรียนได้หลายคน
จังหวัด 1 จังหวัด  →  เกิดอุบัติเหตุได้หลายครั้ง
```

ในข้อมูลอุบัติเหตุไทย:
```
จังหวัดสมุทรสาคร  →  ACC2568000001
                  →  ACC2568000045
                  →  ACC2568000123
                  →  ... (อีกหลายร้อยรายการ)
```

---

### M:N (Many-to-Many) — หลายต่อหลาย

**ความหมาย:** หลายรายการในตาราง A สัมพันธ์กับหลายรายการในตาราง B

**ตัวอย่าง:**
```
นักเรียน M คน ↔ วิชา N วิชา
(สมชายเรียนหลายวิชา และวิชาคณิตฯ มีนักเรียนหลายคนเรียน)
```

> ต้องสร้างตารางกลาง (Junction Table) เพื่อแก้ M:N  
> เช่น ตาราง `enrollments` ที่เชื่อม `students` กับ `subjects`

---

## 📏 Normalization — ทำไมต้องแยกตาราง?

**Normalization** (นอร์มาไลเซชัน) คือกระบวนการแยกข้อมูลออกเป็นตาราง ๆ เพื่อ**ลดการซ้ำซ้อนของข้อมูล** และป้องกันข้อมูลขัดแย้งกัน

### ปัญหาก่อน Normalization (แบบผิด):

```
ตาราง enrollments_bad (ข้อมูลซ้ำซ้อน):
┌─────────┬──────────┬───────────┬──────┬──────────────┬───────┐
│ student │ std_name │ subject   │score │ subj_teacher │ phone │
├─────────┼──────────┼───────────┼──────┼──────────────┼───────┤
│ STU001  │ สมชาย    │ MAT101    │  85  │ ครูสมศักดิ์  │ 081-xx│
│ STU001  │ สมชาย    │ ENG101    │  72  │ ครูสมบัติ    │ 082-xx│  
│ STU001  │ สมชาย    │ SCI101    │  90  │ ครูสมศักดิ์  │ 081-xx│
└─────────┴──────────┴───────────┴──────┴──────────────┴───────┘
ปัญหา: ชื่อสมชายซ้ำ 3 ครั้ง, เบอร์ครูสมศักดิ์ซ้ำ 2 ครั้ง
→ ถ้าเปลี่ยนชื่อ ต้องแก้ทุกแถว → ลืมแก้แถวใดแถวหนึ่ง → ข้อมูลขัดแย้ง!
```

### หลัง Normalization (แบบถูกต้อง):

```
ตาราง students:          ตาราง subjects:         ตาราง enrollments:
┌────────┬──────┐        ┌─────────┬───────────┐  ┌──────────┬─────────┬──────┐
│std_id  │name  │        │subj_id  │subj_name  │  │std_id 🔗 │subj_id🔗│score │
├────────┼──────┤        ├─────────┼───────────┤  ├──────────┼─────────┼──────┤
│STU001  │สมชาย │        │MAT101   │คณิตศาสตร์ │  │STU001    │MAT101   │  85  │
│STU002  │สมหญิง│        │ENG101   │ภาษาอังกฤษ │  │STU001    │ENG101   │  72  │
└────────┴──────┘        └─────────┴───────────┘  │STU001    │SCI101   │  90  │
                                                   └──────────┴─────────┴──────┘
✅ ชื่อสมชายปรากฏแค่ครั้งเดียว
✅ แก้ไขที่เดียว มีผลทั้งระบบ
✅ ไม่มีข้อมูลขัดแย้ง
```

### 3 ระดับของ Normalization (รู้จักเบื้องต้น)

| ระดับ | ชื่อ | กฎหลัก |
|------|------|--------|
| 1NF | First Normal Form (เฟิร์ส นอร์มัล ฟอร์ม) | แต่ละช่องมีค่าเดียว ไม่มีค่าซ้อนกัน |
| 2NF | Second Normal Form (เซ็คัน นอร์มัล ฟอร์ม) | ผ่าน 1NF + แอตทริบิ้วท์ขึ้นกับ Primary Key ทั้งหมด |
| 3NF | Third Normal Form (เทิร์ด นอร์มัล ฟอร์ม) | ผ่าน 2NF + ไม่มีการขึ้นต่อกันระหว่างแอตทริบิ้วท์ที่ไม่ใช่คีย์ |

## 🔧 Workshop 1.3 — ออกแบบ ERD ระบบร้านค้า

**โจทย์:** ร้านขายของชำ "ร้านป้าแดง" ต้องการระบบฐานข้อมูลเก็บข้อมูลดังนี้:
- รายการสินค้า (ชื่อ, ราคา, ประเภท, จำนวนในสต็อก)
- ลูกค้า (ชื่อ, เบอร์โทร, ที่อยู่)
- ใบสั่งซื้อ (วันที่, ลูกค้า, รายการสินค้า, จำนวน)

**ตาราง ERD ที่ออกแบบได้:**

```
┌────────────┐     ┌────────────┐     ┌────────────┐
│  customers │     │   orders   │     │  products  │
│  (ลูกค้า)  │     │ (ใบสั่งซื้อ) │    │  (สินค้า)  │
├────────────┤     ├────────────┤     ├────────────┤
│cust_id 🔑  │─┐   │order_id 🔑 │   ┌─│product_id🔑│
│cust_name   │ │   │cust_id 🔗  │◄──┘ │prod_name   │
│phone       │ └──►│order_date  │     │price       │
│address     │     │total_price │     │category    │
└────────────┘     └────────────┘     │stock_qty   │
                        │             └────────────┘
                        │ 1:N
                        ▼
                ┌────────────────┐
                │  order_items   │
                │ (รายการสินค้า) │
                ├────────────────┤
                │item_id 🔑      │
                │order_id 🔗     │
                │product_id 🔗   │
                │quantity        │
                │unit_price      │
                └────────────────┘
```

**SQL สร้างตาราง:**
```sql
-- สร้างตารางลูกค้า
CREATE TABLE customers (
    cust_id   INTEGER PRIMARY KEY AUTOINCREMENT,
    cust_name TEXT    NOT NULL,
    phone     TEXT,
    address   TEXT
);

-- สร้างตารางสินค้า
CREATE TABLE products (
    product_id INTEGER PRIMARY KEY AUTOINCREMENT,
    prod_name  TEXT    NOT NULL,
    price      REAL    NOT NULL,
    category   TEXT,
    stock_qty  INTEGER DEFAULT 0
);

-- สร้างตารางคำสั่งซื้อ
CREATE TABLE orders (
    order_id   INTEGER PRIMARY KEY AUTOINCREMENT,
    cust_id    INTEGER NOT NULL REFERENCES customers(cust_id),
    order_date DATE    NOT NULL,
    total_price REAL
);

-- สร้างตารางรายละเอียดสินค้าในคำสั่งซื้อ
CREATE TABLE order_items (
    item_id    INTEGER PRIMARY KEY AUTOINCREMENT,
    order_id   INTEGER NOT NULL REFERENCES orders(order_id),
    product_id INTEGER NOT NULL REFERENCES products(product_id),
    quantity   INTEGER NOT NULL,
    unit_price REAL    NOT NULL
);
```

## 💡 Query แรกของเราบนข้อมูลจริง

ลองสำรวจข้อมูลอุบัติเหตุด้วย SQL Query ง่ายๆ:

```sql
-- สถิติภาพรวม
SELECT 
    COUNT(*) AS จำนวนอุบัติเหตุ_ครั้ง,
    SUM(จำนวนผู้เสียชีวิต) AS ผู้เสียชีวิต_ราย,
    SUM(จำนวนผู้บาดเจ็บ) AS ผู้บาดเจ็บ_ราย
FROM road_accidents;

-- ผลลัพธ์ที่ควรได้:
-- จำนวนอุบัติเหตุ_ครั้ง | ผู้เสียชีวิต_ราย | ผู้บาดเจ็บ_ราย
-- 20000                  | ~471              | ~20573
```

```sql
-- ยานพาหนะที่เกิดเหตุมากที่สุด
SELECT ประเภทยานพาหนะหลัก, COUNT(*) AS จำนวน
FROM road_accidents
GROUP BY ประเภทยานพาหนะหลัก
ORDER BY จำนวน DESC;

-- ผลที่คาดว่าจะได้:
-- รถจักรยานยนต์ มากที่สุด (~72% ของทั้งหมด)
```

> **ข้อสังเกต:** ในปี 2568 รถจักรยานยนต์ยังคงเป็นยานพาหนะที่เกิดอุบัติเหตุมากที่สุดในไทย  
> ข้อมูลนี้สำคัญมากสำหรับการกำหนดนโยบายความปลอดภัยทางถนน
